# 前端异常监控及处理

# 前端必要的异常监控

  • JavaScript 脚本异常：这种异常一般是我们写代码时造成的，比如出现语法、逻辑错误等，这种错误也比较容易发现

  • 静态资源加载异常：这种通常是找不到静态资源（image、css、js）或文件加载失败造成的

  • Promise 错误处理异常：这种通常是因为没有catch掉Promise 的错误引起的

# 异常监控方法汇总

  • try-catch： 监控JS执行中的同步代码异常

  • window.onerror： 全局监控 JS里执行中的同步和异步代码异常

  • window.addEventListener： 全局监控静态资源异常

  • unhandledrejection: 全局捕获没有catch的Promise的异常

  • window.load 和 window.beforeunload： 监控网页崩溃

  • Vue errorHandler 和 React componentDidCatch

# JavaScript 脚本异常监控

JavaScript 脚本异常大致包括：

  • 语法错误
  • 运行时出现的同步错误和异步错误

语法错误我们一般在开发阶段就能够发现了，下面主要说一下运行时出现的同步错误及异步错误

# try-catch

try-catch可以帮我们捕获到运行时的 同步错误，如：

try {
  let name = 'lily'
  // _name 未定义
  console.log(_name)
} catch (error) {
  console.log(`错误被捕获：${error}`)
}

同步错误被成功捕获，输出：错误被捕获：ReferenceError: _name is not defined

catch块接收一个包含错误信息的对象，我们可以获取它的message属性来展示错误信息，如：

try {
  // ...
} catch (error) {
  console.log(`错误被捕获：${error.message}`) // 错误被捕获：_name is not defined
}

注意：

try-catch 只能捕获到同步的运行时错误，无法捕获语法错误及异步错误

# window.onerror

window.onerror比try-catch功能要强大一点，它可以同时捕获 同步错误 和 异步错误，而且还可以针对 全局 来捕捉这些错误

/**
 * @param msg {String}：错误消息
 * @param url {String}：引发错误的脚本的 URL
 * @param line {Number}：发生错误的代码行
 * @param colunm {Number}：发生错误的代码列
 * @param error {object}：错误对象
 */
window.onerror = function(message, source, lineno, colno, error) { ... }

注意

window.onerror 要写在其他 js 脚本前面，否则可能会捕获不到错误

另外 window.onerror 是无法捕获到网络异常的错误

# 捕获同步错误：

// 捕获异常
window.onerror = function (msg, url, row, col, error) {
  console.log('捕获到异常了：', {
    msg, url, row, col, error
  })
};

// 同步运行代码测试
let name = 'lily'
// _name 未定义
console.log(_name)

同步运行时错误被捕获到了，结果如下：

捕获到异常了： {msg: "Script error.", url: "", row: 0, col: 0, error: null}

Uncaught ReferenceError: _name is not defined
  at index.js:209

我们看到虽然捕获到了异常，但是浏览器中异常还是会向上抛出Uncaught ReferenceError: _name is not defined。想要避免这种现象只需在window.onerror 函数中return true

# 捕获异步错误：

// 捕获异常
window.onerror = function (msg, url, row, col, error) {
  console.log('捕获到异常了：', {
    msg, url, row, col, error
  })
  return true
};

setTimeout(() => {
  throw new Error('报错啦~')
})

输出：捕获到异常了：{msg: "Script error.", url: "", row: 0, col: 0, error: null}

# 静态资源加载异常监控

通过在window.addEventListener中监听error事件，可以监控静态资源访问失败的情况。

场景：假如我们的静态资源放到了CDN，用户访问的时候资源加载失败了但是我们不知道的情况下就很尴尬了，所以静态资源的监测还是很有必要的。但是这种方式虽然可以捕捉到网络请求的异常，但是无法判断HTTP的状态码，因此还需要配合服务端日志来进行排查

<script>
  window.addEventListener('error', (msg, url, row, col, error) => {
    console.log(`哎呀~静态资源访问不到啦~`, { msg, url, row, col, error })
    return true;
  }, true);
</script>
<!-- common.js、图片都不存在 -->
<img src="./404.png" />
<script src="./common.js"></script>

输出：哎呀~静态资源访问不到啦~ {msg: Event, url: undefined, row: undefined, col: undefined, error: undefined}

# Promise 错误处理异常监控

我们在使用Promise的时候，如果抛出了错误但是忘记显式的通过catch来捕获它的话（该错误无法通过onerror或try-catch捕获），可以通过window.addEventListener监听unhandledrejection事件对该类型的错误进行全局监控

// 监听错误
window.addEventListener("unhandledrejection", function (e) {
  // 阻止浏览器中异常向上抛出
  e.preventDefault()
  console.log(`unhandledrejection:`, e.reason)
  return true
})

// 第一种：Promise.reject
Promise.reject(`Promise.reject 抛出错误~`)

// 第二种：new Promise 中 reject
new Promise((resolve, reject) => {
  reject(`new Promise 中通过 reject 抛出错误~`)
})

// 第三种：throw
new Promise((resolve) => {
  resolve()
}).then(() => {
  throw '通过 throw 抛出的错误~'
})

# 异常上报

监控到错误信息之后就需要把这些信息登记在信息收集平台上，常用的发送形式是如下：

  1. 通过AJAX等发送数据到后端，这种方式存在跨域的风险（不推荐）

  2. 动态创建img标签，这种方式不会跨域，如下：

function report(error) {
  var reportUrl = 'http://xxxx/report';
  new Image().src = reportUrl + 'error=' + error;
}
